# ADR-003: Data Source Abstraction with Optional Firebase Backend

- Status: Proposed
- Date: 2026-03-05
- Deciders: Project maintainers
- Related: `ADR.md`, `ADR-002.md`

## Context

The project currently ships with local static content and includes a Firebase dependency that is not yet used by the application runtime.

Without a clear abstraction, future backend adoption can leak implementation details into UI components and make local development harder.

## Decision

Adopt a repository-style data access abstraction so the UI depends on interfaces, not concrete storage.

### Architecture decisions

1. Define a `contentRepository` interface with read operations such as:
   - `getVideos()`
   - `getBlogPosts()`
   - `getBlogPostBySlug(slug)`
2. Provide a default local implementation (`localContentRepository`) backed by static files.
3. Add an optional Firebase implementation (`firebaseContentRepository`) behind the same interface.
4. Select repository implementation via configuration (environment variable or app bootstrap setting).
5. Keep presentation components unaware of Firebase SDK types and query logic.

## Consequences

### Positive

- UI remains stable if storage backend changes.
- Local-first development stays simple and reliable.
- Backend integration can be rolled out incrementally.

### Negative

- Additional module boundaries and boilerplate are required.
- Team must maintain parity between repository implementations.

## Implementation Plan

1. Create repository interface and local implementation.
2. Refactor `VideoList` and `Blog` data-loading paths to use repository APIs.
3. Add unit tests for repository contracts.
4. Introduce Firebase adapter only when dynamic publishing or remote content is required.

## Review Trigger

Promote this ADR from `Proposed` to `Accepted` when at least one production feature requires remote content updates without redeploying the app.
