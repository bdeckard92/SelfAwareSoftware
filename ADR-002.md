# ADR-002: Structured Content Model and Rendering Strategy

- Status: Accepted
- Date: 2026-03-05
- Deciders: Project maintainers
- Related: `ADR.md`

## Context

Current blog and video content lives in utility files. Blog records include JSX content bodies, and parts of the UI infer metadata by traversing React element internals.

This creates tight coupling between content shape and component implementation, which increases refactor risk and makes non-UI content workflows harder.

## Decision

Adopt a structured content model that separates domain data from rendered components.

### Content model decisions

1. Blog records must include explicit metadata fields:
   - `id` (or `episode`)
   - `title`
   - `subtitle`
   - `dateCreated`
   - `slug`
   - `tags` (optional)
   - `body` (serialized source, not JSX object graph)
2. Video records must include explicit metadata fields:
   - `id`
   - `title`
   - `datePublished`
   - `embedUrl`
   - `tags` (optional)
3. UI components may render content, but must not parse React internals to infer domain fields.
4. Search and sort logic operates only on explicit, typed fields.

## Consequences

### Positive

- Clear data contracts make feature development and testing easier.
- Content can be managed independently from component structure.
- Better compatibility with future API/CMS integration.

### Negative

- Migration effort from JSX-rich entries to structured records.
- Need to define a rendering adapter for serialized content bodies.

## Implementation Notes

- Introduce a `content` module (for example `src/content/`) to hold normalized content sources.
- Introduce adapters/selectors in `src/utils/` or `src/services/` for component consumption.
- Add invariant tests for content schemas and required fields.

## Exit Criteria

- No domain logic depends on `element.props.children` traversal.
- Blog subtitle comes from `subtitle` field.
- All blog/video entries conform to documented fields.
